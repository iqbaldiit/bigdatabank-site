Introduction

What is dbt?

The transformation layer in your modern data stack. It turns SQL into reusable, testable, version-controlled code.

ELT, Not ETL
dbt does the Transform only. Your data warehouse handles Extract & Load. dbt runs SQL directly inside the warehouse.
Version Controlled
Every transformation is a .sql file in Git. Full CI/CD, code review, and audit trail — like software engineering for data.
DAG by Convention
Dependencies are inferred via ref() and source(). No manual DAG configuration — the graph builds itself.

Key Insight: dbt does not move or store data. It transforms data in place within your warehouse using SQL (and optionally Python).

Layering Convention

Staging View

Raw source cleanup. Rename columns, cast types, light filtering. One model per source table.

Intermediate View / Ephemeral

Join staging models, apply business rules. Not consumed directly by BI tools.

Marts Table / Incremental

Dimension and fact models. Aggregated, denormalized. This is what BI tools query.

Snapshots Snapshot

Track changes to source data over time using SCD Type 2 logic.

Rule of thumb: Never reference a staging model from a mart. Always go through intermediate. Keeps the DAG clean.

Core Terminology

Model
A .sql file with a SELECT statement. Becomes a view or table in your warehouse.
ref()
Jinja function to reference another model. {{ ref('stg_orders') }} — dbt resolves the dependency.
source()
Jinja function to reference raw data. Defined in schema.yml. Enables freshness checks.
Seed
A .csv file in seeds/. Loaded into warehouse as a table. For static reference data.
Test
Assertions: unique, not_null, accepted_values, relationships. Custom tests via macros.
Exposure
Declare downstream consumers (dashboards, notebooks) in schema.yml. Enables impact analysis.

SCD Type 2 with dbt Snapshots

Track historical changes — when a customer changes email, keep both old and new with valid_from/valid_to windows.

How dbt Snapshots Work

1

Check Source

Compare current source against last snapshot using strategy and updated_at.

2

Detect Changes

If tracked column changed, close old record (valid_to = now, is_current = false).

3

Insert New Row

New row with valid_from = now, valid_to = null, is_current = true.

4

New Records

If record doesn't exist in snapshot yet, insert as brand new row.

sql — snapshots/snap_customers.sql {% snapshot snap_customers %} {{ config( target_schema='snapshots', strategy='timestamp', updated_at='updated_at', unique_key='customer_id', invalidate_hard_deletes=true ) }} SELECT customer_id, customer_name, email, city, updated_at FROM {{ source('raw', 'customers') }} {% endsnapshot %}

strategies: timestamp — uses updated_at column. check — checks specific columns: check=['email', 'city'].

The Late Arrival Problem

Event data arrives after the fact — e.g., a transaction from 3 days ago appears today. Your incremental model already processed up to "yesterday."

Late Arrival Solutions

1. Lookback Window

Use updated_at >= max - interval '3 days'. Re-process overlapping period. Deduplicate with QUALIFY ROW_NUMBER()....

2. Source Watermark Table

Maintain a control table tracking high-water-mark per source. Reset when late data is detected.

3. Full Refresh + Partition

Periodic full refreshes (weekly) on a partition. Incremental daily, full refresh weekends.

4. Event-Time Processing

Use event's own timestamp. Build with configurable window in dbt_project.yml.

sql — Lookback Window Pattern {% set lookback_hours = var('lookback_hours', 72) %} SELECT * FROM {{ ref('stg_events') }} WHERE event_timestamp >= ( SELECT MAX(event_timestamp) FROM {{ this }} ) - INTERVAL '{{ lookback_hours }} hours' QUALIFY ROW_NUMBER() OVER( PARTITION BY event_id ORDER BY loaded_at DESC ) = 1

Macros & Jinja Reusability

Stop copy-pasting SQL. Build reusable macros for common patterns.

sql — macros/generate_surrogate_key.sql {% macro generate_surrogate_key(columns) %} MD5( CONCAT_WS('|', {% for col in columns %} COALESCE(CAST({{ col }} AS VARCHAR), '') {% if not loop.last %}, {% endif %} {% endfor %} ) ) {% endmacro %}
sql — macros/pivot_statuses.sql {% macro pivot_statuses(source_table, pivot_col) %} SELECT order_id, MAX(CASE WHEN {{ pivot_col }} = 'created' THEN 1 ELSE 0 END) AS was_created, MAX(CASE WHEN {{ pivot_col }} = 'shipped' THEN 1 ELSE 0 END) AS was_shipped FROM {{ source_table }} GROUP BY order_id {% endmacro %}
sql — Using the macro SELECT {{ generate_surrogate_key( ['customer_id', 'order_date', 'product_id'] ) }} AS order_line_key, customer_id, order_date, product_id, quantity, revenue FROM {{ ref('stg_order_lines') }}

When to Use Macros

Surrogate key generation
Date spine / calendar tables
Pivot/unpivot patterns
Conditional logic per warehouse
Custom test assertions
Graduated calculations

dbt-utils already includes generate_surrogate_key, date_spine, pivot. Check before writing your own!

Edge Cases

Late-Arriving Partitions

Pass partitions via --vars. insert_overwrite replaces only those partitions.

Duplicates Within Partition

Add QUALIFY ROW_NUMBER() OVER(...) = 1 to deduplicate.

Schema Changes

on_schema_change = 'append_new_columns' or 'fail'.

Full Refresh Fallback

Test with dbt run --full-refresh before deploying changes.

Custom Materialization

sql — macros/materializations/clone.sql {% materialization clone, default %} {% set relation = create_or_replace_relation(this) %} {% set source = ref('source_model') %} {% do %} {% call statement('main') -%} CREATE OR REPLACE TABLE {{ relation }} CLONE {{ source }} {%- endcall %} {% enddo %} {{ return({'relations': [relation]}) }} {% endmaterialization %}

When to write custom materializations: Warehouse-specific features (Snowflake Clones, BigQuery ML), non-standard patterns, cross-db replication.

CI/CD & Best Practices

CI Pipeline (on PR)

1

dbt compile

Verify Jinja renders. Fast — no warehouse query.

2

dbt lint

Naming conventions, unused models, circular deps.

3

dbt build --select state:modified+

Run + test only changed models and downstream.

4

Artifact Comparison

Compare manifest.json for breaking changes.

CD Pipeline (on merge)

dbt run --select +fct_* +dim_*
dbt test --select fct_* dim_*
dbt source freshness
dbt docs generate & deploy
Notify Slack on failure

Best Practices

Naming Conventions

stg_, int_, fct_, dim_, snap_. Always prefix with layer.

One Model, One Purpose

Split into composable pieces. Easier testing and reuse.

No SELECT *

Explicitly list columns. Self-documenting, prevents surprise breaks.

YAML for Metadata

Tests, descriptions, owners — all in YAML. Powers docs site.

Environments

dev (personal), ci (PR), prod (scheduled).

Never Hardcode

Use var(), env_var(), target.schema.
